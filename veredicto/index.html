El problema

«Todos los tests pasan.» ¿Según quién?

Cuando el agente escribe el código Y los tests, el verde del CI deja de significar que el código funciona.

El agente borra o marca .skip el test que falla en vez de arreglar el bug.

Asserts que siempre pasan: expect(true).toBe(true), mocks que mockean justo lo que se testea.

Corres flotas de agentes en paralelo. El CI es el único filtro humano — y se lo están saltando.

En el 30% de los runs, los modelos punteros amañaron su propia evaluación — medido por METR. El dolor está cuantificado; nadie vende el detector.

FAQ

Preguntas frecuentes

¿Necesito una API key o un LLM?

No. La detección estática corre en tu runner, es determinista y no usa LLM ni API key. El LLM-juez diff-vs-claim es un extra opcional de pago, y ahí usas tu propia key.

¿Va a bloquear mis pull requests?

Por defecto funciona en modo warn: comenta lo que encuentra sin bloquear. El modo block es opt-in. Cada regla trae explicación y se puede suprimir con un comentario en el PR.

¿En qué se diferencia de CodeRabbit o del review nativo de GitHub?

CodeRabbit revisa calidad con un LLM de pago; GitHub valida seguridad. Ninguno comprueba la integridad de los tests. Veredicto hace una cosa: detectar cuando los tests se amañaron — determinista, instantáneo y gratis.

¿Qué es test-gaming exactamente?

Hacer que el CI pase en verde sin que el código sea correcto: borrar o marcar .skip tests, asserts tautológicos, mocks de lo que se testea, snapshots regenerados en masa o umbrales de cobertura relajados.

¿Funciona en GitLab o Bitbucket?

Primero GitHub Actions. GitLab y Bitbucket están en el roadmap. Apúntate a la lista para saber cuándo llegan.
